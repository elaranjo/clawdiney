# Proposal: project-knowledge-graph

## Why

Clawdiney's graph only knows what the user manually linked (WikiLinks, tags). It cannot answer the questions that matter across a multi-project workspace: "how do my projects relate?", "which projects use pattern X?", "what does service A consume from service B?". The v0.2.0 schema (`entities.kind`, `relations.confidence`, `evidence_chunk_id`) was designed for typed cross-project knowledge but nothing populates it yet. Meanwhile `project_indexer.py` already generates per-project docs — the natural substrate to enrich.

## What Changes

- Enrich project cards generated by `project_indexer.py`: add **Purpose** and **Architecture** sections (LLM-generated from README + structure via Ollama) and an **Interfaces** section (exposes/consumes: ports, APIs, datastores, env-var connections) parsed deterministically.
- Add deterministic entity extraction (layer 1): parse manifests (`pyproject.toml`, `package.json`, `docker-compose.yml`, `.env.example`) into typed entities (`project`, `library`, `service`, `datastore`) and relations (`DEPENDS_ON`, `SHARES_DB`, `CALLS_API_OF`) with `confidence=1.0`.
- Add LLM entity extraction (layer 2): structured JSON extraction over project cards producing `USES_PATTERN` / `IMPLEMENTS` / `MENTIONS` relations with model confidence and chunk evidence; embedding-similarity entity resolution to merge duplicates ("JWT" vs "jwt auth").
- Add MCP tools: `get_project_card(name)` (full card retrieval) and `how_do_projects_relate(a, b)` (graph paths between two projects with relation types and evidence).
- Extend `explore_graph` output to show typed relations with evidence for non-note entities.

## Capabilities

### New Capabilities
- `project-cards`: Enriched per-project Markdown cards (Purpose, Stack, Architecture, Interfaces, Conventions, Entry points) generated into the vault and indexed like any note.
- `entity-extraction`: Two-layer extraction pipeline populating typed entities/relations in brain.db — deterministic manifest parsing (confidence 1.0) and LLM-based semantic extraction (scored confidence, evidence-linked), with entity resolution to prevent duplicates.
- `project-graph-tools`: MCP tools for cross-project knowledge — project card retrieval and project-to-project relationship paths with evidence.

### Modified Capabilities
- `graph-store`: Multi-hop traversal and related-entity queries extended to cover non-note entity kinds (`project`, `service`, `library`, `pattern`, `datastore`) and to return relation evidence.

## Impact

- **Code**: `project_indexer.py` (card sections), new `entity_extractor.py` (both layers), `storage.py` (entity kinds, path queries, evidence joins), `mcp_server.py` (2 new tools, `explore_graph` output), `scripts/watch_projects.py` (trigger extraction on card change).
- **Dependencies**: none new — LLM extraction uses the existing Ollama client; entity resolution uses the existing embedding provider.
- **Storage**: no schema migration needed (fields already exist); new rows only.
- **MCP API surface**: additive — existing tool signatures unchanged.
- **Cost/performance**: layer 2 runs only when a project card's content hash changes (~1 LLM call per project change); layer 1 is parse-only.
